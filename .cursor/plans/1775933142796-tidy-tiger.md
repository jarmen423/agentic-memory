# OpenClaw Production-Readiness Blueprint Plan

## Goal

Produce a repo-native readiness blueprint that takes the current OpenClaw integration from "working prototype" to "production-ready product surface" with:

- frictionless install for OpenClaw users
- reliable centralized memory for 10 concurrent OpenClaw agents
- a premium dashboard / control-plane UX
- concrete architecture, operations, testing, security, packaging, and GTM deliverables

This plan is for creating the blueprint deliverable set, not for implementing the platform changes yet.

## Planning Constraints

- Plan mode is read-only except for this file.
- The plan must stay grounded in the current repository state.
- The next execution phase should mainly author blueprint docs, templates, and supporting artifacts; any later code changes should follow the blueprint.

### Key gaps that the blueprint must address

- No production-grade distribution path for the OpenClaw plugin:
  - `packages/am-openclaw/package.json` is still `private: true`
- No Node/plugin CI or release workflow:
  - `.github/workflows/ci.yml` is Python-only
- Desktop shell is a thin local web shell, not a signed desktop product:
  - `desktop_shell/static/index.html` still contains placeholder cards
- OpenClaw runtime tests are mostly backend-side; the plugin package lacks direct tests
- Canonical reads are real for conversation turns but partial for non-conversation hits
- Product state is persisted locally in JSON, which is not sufficient as the production control-plane source of truth
- There is no full install -> login -> plugin activation -> dashboard onboarding journey yet
- There is no formal performance, load, chaos, security, or operational readiness program for OpenClaw

## Recommended Defaults To Bake Into The Blueprint

These defaults remove ambiguity for the implementation phase.

1. Deployment model
   - Default to a hosted multi-tenant control plane plus a packaged local desktop companion.
   - Preserve a self-hosted path as a secondary deployment profile, not the primary UX.

2. Runtime topology
   - Keep `am-server` as the initial API boundary for beta.
   - Add durable asynchronous ingestion and centralized control-plane storage instead of prematurely splitting into many microservices.
   - Treat SpacetimeDB/temporal capabilities as additive, not GA-critical, unless validation proves otherwise.

3. Install experience
   - The long-term install path should not require users to manually supply `backendUrl` and raw API keys.
   - The desktop companion should own auth, config bootstrap, health checks, update prompts, and plugin activation.

4. Auth model
   - Use user login plus workspace-scoped short-lived tokens for the local companion and plugin.
   - Keep static bearer-token auth only as a dev/self-hosted fallback.

5. Control-plane persistence
   - Move production session, project, integration, audit, and onboarding state to a server-side relational store.
   - Keep the current local JSON product state as a cache / offline diagnostic artifact only.

6. Packaging strategy
   - Publish the OpenClaw plugin as a versioned distributable package.
   - Package the desktop shell as a signed desktop app with auto-update.
   - Keep CLI-first developer flows for local development.

7. Scope target
   - v1 production target is 10 concurrent OpenClaw agents per workspace with clear headroom to 25.

## Target Architecture Direction To Document

The blueprint should converge on the following target state.

### Recommended product architecture

- `OpenClaw Plugin` (`am-openclaw`)
  - memory runtime
  - context-engine hooks used for capture and optional augmentation
  - local fail-safe buffering hooks and health reporting
- `Desktop Companion`
  - packaged control plane / dashboard
  - login, workspace selection, configuration bootstrap, diagnostics, installer orchestration, local log access
  - local proxy to hide raw backend credentials from end users
- `Hosted Agentic Memory Control Plane`
  - auth/session/token service
  - workspace/project/integration metadata service
  - OpenClaw API gateway / rate limiting / idempotency
  - async ingestion coordinator
  - search/context/read APIs
- `Storage`
  - Neo4j for shared memory graph and retrieval
  - relational store for workspaces, sessions, projects, onboarding state, audit trails, release channels
  - object storage for exported artifacts, logs, installer manifests, and support bundles
  - optional queue/cache layer for async ingest and rate limiting

### Recommended architectural principles

- Modular monolith first; split services only where justified by load or ownership.
- Queue ingestion, do not block user turns on embeddings/indexing.
- Degrade gracefully: capture > search > context augmentation.
- Identity is `workspace_id + device_id + agent_id + session_id`; project remains session-scoped.
- Observability and audit fields are first-class in every API contract.

### Explicit failure modes the blueprint must cover

- plugin install fails or is incompatible with host version
- backend auth/token expiration
- desktop companion offline / loopback unavailable
- API gateway unavailable or rate-limiting
- ingestion queue backlog or poison messages
- Neo4j latency spikes / partial outage
- canonical read unsupported for a hit type
- context augmentation timeout or oversized response
- duplicate turn ingest / replay / ordering drift
- workspace/project mismatch or session inference failure
- release rollback / auto-updater failure / partial plugin upgrade

## Recommended SLOs And Resource Budgets To Document

These values should be used as the initial blueprint defaults unless later analysis disproves them.

### Beta targets

- 10 concurrent active agents per workspace
- ingest ACK p95: <= 250 ms when accepted into queue
- memory search p95: <= 800 ms
- context resolve p95: <= 1500 ms
- canonical read p95: <= 400 ms for supported hit types
- dashboard page-load p95: <= 2.0 s on desktop broadband
- availability: 99.5%
- RPO: <= 5 minutes
- RTO: <= 60 minutes

### GA targets

- 10 concurrent active agents with 2.5x burst headroom
- ingest ACK p95: <= 150 ms
- memory search p95: <= 600 ms
- context resolve p95: <= 1200 ms
- canonical read p95: <= 300 ms
- dashboard page-load p95: <= 1.5 s
- availability: 99.9%
- RPO: <= 1 minute
- RTO: <= 15 minutes

### Initial production sizing for the blueprint

- API: 2 replicas, each 2 vCPU / 4 GB RAM
- worker: 2 replicas, each 2 vCPU / 4 GB RAM
- Neo4j: managed or self-hosted equivalent of 4 vCPU / 16 GB RAM / SSD-backed storage
- relational store: 2 vCPU / 4 GB RAM
- queue/cache: 1 vCPU / 2 GB RAM
- desktop companion target footprint: < 250 MB RAM steady-state on a typical dev machine

## Recommended UX Direction To Document

The blueprint should not preserve the current placeholder-heavy shell UX as the target state.

### UX north star

- install in minutes, not hours
- clear "connected / indexing / healthy / degraded" states
- fleet view for 10 OpenClaw agents
- memory activity, queue health, latency, and recent failures in one place
- guided onboarding and recovery steps instead of raw JSON

### Dashboard structure to specify

- onboarding rail
- workspace health hero
- agent fleet table / cards
- memory throughput and recall quality charts
- integration health matrix
- incidents and alerts drawer
- configuration / project / retention settings
- support bundle export

### Design system direction to specify

- upscale neutral palette with restrained accent colors; no default purple bias
- typography: `Manrope` for UI, `IBM Plex Mono` for operational data, optional editorial accent `Fraunces` or `Newsreader`
- 8px spacing scale
- strong empty states, status semantics, and accessibility-first contrast
- component library guidance for cards, tables, charts, filters, drawers, banners, and install wizard steps

## Recommended GTM Direction To Document

- Positioning: "Shared memory and operational control plane for OpenClaw power users and agent teams."
- Packaging:
  - Community / self-hosted dev path
  - Pro workspace plan including hosted control plane and desktop app
  - Team / Enterprise plan with SSO, audit export, retention controls, support SLA
- Launch motion:
  - private alpha with dogfood + design partners
  - public beta with install telemetry and white-glove onboarding

## Release Gates To Encode In The Blueprint

### Public beta gate

- plugin install success >= 85% on supported hosts
- first-memory-success within 15 minutes for >= 70% of new beta workspaces
- 10-agent shared-memory soak test passes for 24 hours without data loss
- no open critical security findings
- p95 API targets meet beta thresholds for 7 consecutive days

### GA gate

- plugin install success >= 95%
- first-memory-success within 10 minutes for >= 85% of new workspaces
- 10-agent soak test passes for 7 consecutive days
- rollback drill completed successfully
- audit log export, alerting, and support runbooks verified
- p95 API targets meet GA thresholds for 14 consecutive days

## Major Risks To Capture In The Blueprint

| Risk | Impact | Mitigation |
| --- | --- | --- |
| OpenClaw host lifecycle constraints still require context-engine slot for capture | UX and mental-model confusion | document clearly, test heavily, engage host roadmap, isolate in setup UX |
| Plugin package lacks direct tests today | regressions reach users | add Node/plugin CI, mocks, contract tests, host-installed e2e tests |
| Local JSON state is not production-safe | lost project/session/audit state | migrate production source of truth to relational store |
| Manual API-key setup creates install friction | low activation and support burden | move to desktop-managed auth and bootstrap |
| Canonical reads remain partial | poor retrieval trust | define staged read-contract roadmap and fallback behavior |
| No signed installer or auto-update path | poor adoption and high support cost | package desktop companion with release channels and rollback plan |
| No Node release automation | plugin distribution becomes fragile | add plugin build/test/publish/release workflows |
| Neo4j becomes bottleneck under concurrent agents | latency and degraded recall | add queueing, caching, query budgets, and vertical scaling plan |
| Missing security and audit foundations | enterprise blockers | define authn/authz, audit fields, secrets, SBOM, vuln scanning, and log retention |
