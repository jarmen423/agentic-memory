# W14-OC-01 Handoff

## Status

- Completed

## What Changed So Far

- Hardened hosted MCP observability in `src/am_server/metrics.py` by adding
  per-surface request counters for public vs internal MCP mounts.
- Added OpenClaw domain metrics in `src/am_server/metrics.py` for:
  - session registration gauge
  - turn-ingest counters
  - ingest error counters
  - search request counters and latency summaries
  - context-resolve latency summaries
- Added lightweight Phase 14 read-path caching in
  `src/am_server/routes/openclaw.py` for:
  - `/openclaw/project/status` lookups (60s TTL)
  - identical `/openclaw/memory/search` requests (30s TTL)
  - cache invalidation on session registration, project activation/deactivation,
    and turn ingest
- Tuned Neo4j pool acquisition in `src/agentic_memory/core/connection.py` to
  fail fast by default (`10s` instead of `60s`) and exposed pool settings for
  runtime health reporting.
- Updated `src/am_server/app.py` so hosted MCP requests now emit stable
  response headers identifying:
  - the matched MCP surface
  - the auth surface used for that mount
- Published backend runtime component state from `src/am_server/app.py` into the
  local product store so `/openclaw/health/detailed` reflects:
  - server health
  - MCP surface inventory
  - memory pipeline warm-up state
  - context-engine pipeline warm-up state
- Updated `src/am_server/routes/openclaw.py` so successful OpenClaw calls emit
  the new domain metrics at the route boundary.
- Extended `tests/test_am_server.py` to cover:
  - MCP auth-failure headers
  - REST-key fallback when public/internal MCP keys are unset
  - surface-specific rotated public MCP keys
  - Prometheus metrics for MCP surface traffic
  - detailed health runtime component payloads
  - fail-fast Neo4j connection-manager defaults
  - project-status cache reuse and invalidation
  - search cache reuse and invalidation
- Updated `tests/load/test_openclaw_load.py` metric reset logic so the new
  MCP-surface and OpenClaw domain counters do not leak between tests.
- Updated `tests/e2e/test_openclaw_e2e.py` and
  `tests/chaos/test_openclaw_chaos.py` to clear the new metric families
  between runs.

  limiting and deeper write-path pressure controls.
- That item was explicitly deferred by user direction during Phase 14.
- For this wave, backend completion now means:
  - MCP surface boundaries are explicit
  - request/route observability is in place
  - repeated OpenClaw read paths are cached and invalidated correctly
  - Neo4j pool defaults fail fast enough for beta operations
  - runtime component status is visible through the OpenClaw health surface
